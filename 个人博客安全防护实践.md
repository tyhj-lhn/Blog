# 🛡️ 个人博客安全防护实战

> 从 XSS 到 DDoS，一个小破站能做的所有安全加固。

**摘要：** 本文拆解了一个个人博客的纵深防御体系——从输入端的 XSS 过滤与文件上传校验，到认证侧的 JWT 双 Token 轮转与渐进式锁定，再到运维层的速率限制与服务器加固。每一项措施都对应真实攻击场景，并附上具体的实现策略和踩坑记录。

---

当你决定自己搭一个博客，并且允许陌生人留言、评论时，你已经向互联网的暗面打开了一扇门。

即使你的博客每天只有几十个访客，恶意流量也不会放过你——扫描器不分大小站，攻击脚本没有感情。以下是我从零开始做安全防线时，每一层都踩过的坑和最终落地的方案。

---

## 🔤 第一层：输入过滤

### 你收到的每段文字都可能是投毒

前端校验只是用户体验层面的提示（"请输入有效邮箱"）。真正的恶意请求根本不会走你的页面——curl、Postman、自己写的脚本可以绕过一切前端限制，直接向你的 API 扔 payload。

### XSS 防护：空白名单策略

**攻击场景：** 用户在评论中插入 `<script>alert(document.cookie)</script>`。若未经处理直接存库，其他用户浏览评论区时该脚本就会执行，窃取 Cookie、劫持会话。

**常见错误做法：** 用正则匹配黑名单——`<script>`、`onerror`、`javascript:`……总有你没见过的绕过方式。Unicode 同形字符、HTML 实体编码、大小写变体——黑名单几乎不可能穷尽。

**我的方案：**

```typescript
// lib/sanitize.ts
import xss from 'xss';

const XSS_OPTIONS = {
  whiteList: {},             // 空对象 = 不信任任何 HTML 标签
  stripIgnoreTag: true,      // 去掉不允许的标签体
  stripIgnoreTagBody: true,  // 去掉 <script> 标签体内的内容
};

export function sanitizeContent(input: string): string {
  return xss(input, XSS_OPTIONS).trim();
}
```

核心思路就一条：**不保留任何 HTML。** `<b>`、`<img>`、`<a>` ——统统过滤。同时 `trim()` 去掉首尾空白，防止 Unicode 同形字符注入。所有用户输入点——评论内容、用户名、留言昵称、留言内容——统一在写入数据库之前走 `sanitizeContent()`。

### 为什么不做富文本留言

每多一个 HTML 标签，就多一个攻击面。对于个人博客，纯文本留言足够。简洁本身也是体验。

## 🔐 第二层：认证安全

### 最薄弱的环节往往是口令

#### JWT 双 Token 轮转

| Token | 有效期 | 用途 |
|-------|--------|------|
| Access Token | 15 分钟 | 每次请求携带 `Authorization: Bearer <accessToken>` |
| Refresh Token | 7 天 | 仅用于 `/api/auth/refresh` 换取新 Token 对 |

**设计思路：**

- Access Token 短命（15min）：即使泄露，攻击窗口极短
- Refresh Token 长命但只用一次：用户不需要频繁输入密码
- 前端 401 自动刷新队列：多个请求同时过期 → 只发一次 refresh → 所有请求共享新 token

```typescript
// 前端 401 刷新队列（简化）
let refreshPromise: Promise<string | null> | null = null;

async function refreshAccessToken(): Promise<string | null> {
  if (!refreshPromise) {
    refreshPromise = fetch('/api/auth/refresh', {
      method: 'POST',
      body: JSON.stringify({ refreshToken: getRefreshToken() }),
    })
      .then(r => (r.ok ? r.json() : null))
      .finally(() => { refreshPromise = null; });
  }
  return refreshPromise;
}
```

多个 API 调用同时收到 401 → 只有一个会发起真实的 refresh 请求，其余的 `await` 同一个 Promise。

#### Token 即时撤销

**攻击场景：** 管理员在公共电脑上忘了登出，Refresh Token 泄露。改密码之后，攻击者手上的 Refresh Token 还有 7 天有效期——如果服务端不做额外校验，攻击者可以一直刷新。

**实现：**

```prisma
model User {
  tokenVersion Int @default(0)
}
```

改密码时 `tokenVersion` 原子递增（用 Prisma 的 `{ increment: 1 }`，不用 `tokenVersion + 1`，防止并发 TOCTOU 竞态）：

```typescript
await prisma.user.update({
  where: { id: user.id },
  data: { tokenVersion: { increment: 1 } },  // 原子递增
});
```

JWT payload 里嵌入 `tokenVersion`。刷新时比对：

```typescript
if (decoded.tokenVersion !== user.tokenVersion) {
  throw unauthorized('Token 已失效，请重新登录');
}
```

效果：改密码 → 版本号 +1 → 所有设备上的所有旧 token **瞬间全部失效**。

#### 密码存储

- bcryptjs，cost 因子 12（单次哈希约 250ms，充分对抗离线暴力破解）
- 密码字段不出现在任何日志中
- 传输层：Nginx TLS 终止，明文密码永不流经公网

## 🚪 第三层：暴力破解防护

### 让攻击者知难而退

**攻击场景：** 攻击者用 `rockyou.txt`（1400 万个常见密码）对 `admin@memorystory.dev` 持续尝试。

**需求矛盾：** 一次失败就锁 → 真实用户被误伤。无限尝试 → 密码早晚被撞开。需要渐进式惩罚。

```typescript
// lib/login-guard.ts 核心逻辑
const TIER_1 = { limit: 5,  lockoutMs: 15 * 60 * 1000 };  // 5 次失败 → 15 分钟
const TIER_2 = { limit: 10, lockoutMs: 60 * 60 * 1000 };  // 10 次失败 → 1 小时
const RESET_WINDOW = 30 * 60 * 1000;                       // 30 分钟无失败 → 自动重置

function checkLockout(email: string): void {
  const record = lockoutMap.get(email);
  if (!record) return;

  const lockTime = record.failures >= TIER_2.limit
    ? TIER_2.lockoutMs
    : record.failures >= TIER_1.limit
      ? TIER_1.lockoutMs
      : 0;

  if (lockTime > 0) {
    const remaining = record.lastAttempt.getTime() + lockTime - Date.now();
    if (remaining > 0) {
      throw tooManyRequests(`请 ${Math.ceil(remaining / 60000)} 分钟后再试`);
    }
  }
}
```

**防用户枚举 —— 关键细节：**

```typescript
// 用户不存在 → 也记录失败 ← 防止攻击者通过"是否存在"筛选有效邮箱
if (!user) {
  await recordFailedAttempt(email);
  throw unauthorized('邮箱或密码错误');  // 与密码错误的响应完全相同
}

// 密码错误
await recordFailedAttempt(email);
throw unauthorized('邮箱或密码错误');  // 相同消息、相同状态码、相同耗时
```

攻击者无法判断一个邮箱是否注册——无论哪种情况，响应完全一致。

## 📤 第四层：文件上传安全

### 最容易被忽视的高危攻击面

文件上传是 OWASP Top 10 的常客。我的上传端点只对管理员开放，但防御不做假设。

#### 七重过滤链

| 阶段 | 措施 | 拦截 |
|------|------|------|
| 1 | MIME 白名单 | `text/html`、`application/octet-stream` |
| 2 | 扩展名白名单 | `.php`、`.exe`、`.html`、`.svg`、`.xml` |
| 3 | 魔数校验（文件头 12 字节） | 把 `.php` 改名为 `.jpg` 的伪装文件 |
| 4 | 大小限制（5MB / 50MB 双层） | 超大文件 DoS |
| 5 | UUID 重命名 | 路径注入、文件名冲突 |
| 6 | 路径遍历校验 | `../../../etc/passwd` |
| 7 | 速率限制（20/min） | 洪水攻击 |

#### 魔数校验 —— 真正识别"这是什么文件"

```typescript
const MAGIC_BYTES: Record<string, number[][]> = {
  'image/jpeg': [[0xFF, 0xD8, 0xFF]],                          // JPEG
  'image/png':  [[0x89, 0x50, 0x4E, 0x47]],                    // .PNG
  'video/mp4':  [[0x00, 0x00, 0x00, 0x18, 0x66, 0x74, 0x79, 0x70],  // ...ftyp
                 [0x00, 0x00, 0x00, 0x20, 0x66, 0x74, 0x79, 0x70]], // ...ftyp
};

function validateMagicBytes(buffer: Buffer, mimeType: string): boolean {
  const signatures = MAGIC_BYTES[mimeType];
  if (!signatures) return true; // 未知类型不拒绝（宽容策略）
  return signatures.some(sig =>
    sig.every((byte, i) => byte === 0x00 || byte === buffer[i])
  );
}
```

攻击者把 `evil.php` 改名为 `photo.jpg`？可以改扩展名、伪造 MIME，但二进制特征改不了。读到文件头不是 `FF D8 FF`，直接拒绝。

`0x00` 作为通配符——MP4 的 ISO BMFF 容器前 4 字节是可变大小字段，不是固定值。

#### 路径遍历防护

```typescript
app.delete('/api/admin/uploads/:filename', async (req) => {
  const { filename } = req.params;

  // 第一道：拒绝跳转字符
  if (filename.includes('..') || filename.includes('/') || filename.includes('\\')) {
    throw validationError('非法的文件名');
  }

  const filePath = path.join(UPLOADS_DIR, filename);

  // 第二道：解析后二次确认
  if (!filePath.startsWith(UPLOADS_DIR)) {
    throw forbidden('路径越界');
  }

  await fs.promises.unlink(filePath);
});
```

**两道防线：** 先检查输入是否包含 `..` `/` `\`，再用 `path.join()` 解析后二次确认绝对路径是否仍在 uploads 目录内。后者是兜底——即使第一步有遗漏。

## 🐢 第五层：速率限制

### 慢慢来，大家都安全

```typescript
// middleware/rate-limit.ts
export const rateLimitPresets = {
  global:    { max: 100, timeWindow: 60000 },  // 全局
  auth:      { max: 5,   timeWindow: 60000 },  // 登录 —— 极低
  comment:   { max: 3,   timeWindow: 60000 },  // 评论
  guestbook: { max: 3,   timeWindow: 60000 },  // 留言
  upload:    { max: 20,  timeWindow: 60000 },  // 上传
};
```

不同端点不同限额，各司其职：登录接口 5/min 防止暴力破解，评论/留言 3/min 防灌水，上传 20/min 防洪水，其余走全局 100/min。配合 Fastify 的 `bodyLimit: 512KB` 和 `requestTimeout: 30000`，作为服务器层兜底。

## 🏰 第六层：服务器加固

### 最后一道门

```bash
# ufw —— 只开三个端口
ufw allow 22/tcp         # SSH（仅密钥登录）
ufw allow 'Nginx Full'   # 80 + 443
ufw enable
```

后端只监听 `127.0.0.1:3001`——3001 端口不对公网开放。唯一到达后端的路径是 `Nginx → proxy_pass http://127.0.0.1:3001`。

```nginx
# Nginx 加固
server_tokens off;                                  # 不泄露版本号
client_max_body_size 50m;                           # 请求体上限
add_header Strict-Transport-Security "max-age=..." always;  # HSTS
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
```

Helmet 自动设置安全头，Nginx 层叠加 HSTS 与 `server_tokens off`。

## 🧠 第七层：安全意识

### 工具之外的防线

上面这些措施，大部分 AI 能帮你写。但有几个安全决策，是 AI 帮不了的：

**1. 隐藏式管理后台：** 导航栏没有任何登录链接，`/admin` 是一个"知道它存在才能访问"的路径。这不是真正的安全机制（Security through obscurity），但能筛掉 99.9% 的自动化扫描器。

**2. 生产密钥强制校验：** `validateSecrets()` 在 `NODE_ENV=production` 时拒绝弱 JWT 密钥——如果 `.env` 里还是 `change-me-to-a-random-64-char-string`，直接 `process.exit(1)`。部署者忘记替换密钥是常见事故，代码里主动加这道锁。

**3. 每写一个接口自检三个问题：**

> 🔍 输入验证了吗？
> 🧹 XSS 过滤了吗？
> 🚷 权限检查了吗？

这三个问题比任何工具都管用——它们在代码进入 review 之前、在被部署之前、在攻击者找到漏洞之前拦截问题。

## 💭 最后

安全不是一个"加个 WAF 就完事"的一次性工作。它是一种持续的思维方式——你每次写新功能时，都在攻击面上加了一个点，或者减了。

个人博客不是银行，被攻破的后果没有资金损失，但会有更糟的东西：域名被标为恶意站点、服务器变成挖矿肉鸡、数据被拖库。

**花 10% 的开发时间做安全，可能减少 99% 的潜在灾难。** 而这些实践——XSS 过滤、速率限制、Token 管理、文件上传校验——不会只服务这一个博客，它们是通用的、可迁移的、会伴随你写下一个项目的肌肉记忆。
